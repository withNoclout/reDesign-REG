# บันทึกการดำเนินงานและแผนงาน (Implementation Plan) - 22/02/2026
## เรื่อง: ระบบประเมินอาจารย์ผู้สอน (Teacher Evaluation System)

---

### 📝 รายการที่ดำเนินการไปแล้ว (Log of Actions Done)
1. **[Authentication]**
   - อัปเดตไฟล์ `.env.local` เพื่อรองรับบัญชีทดสอบใหม่ (`s6701091611176`) ที่มีข้อมูลค้างการประเมินจริง
   - แก้ไขระบบการดึง Token ให้รองรับทั้ง `reg4` (Login) และ `reg3/reg2` (API Data)

2. **[Data Infrastructure - Question Fetching]**
   - ค้นพบ Endpoint จริงของมหาวิทยาลัยที่ `https://reg2.kmutnb.ac.th/regapiweb3/api/th/Evaluateofficerform/Evaluatequestion/`
   - แก้ไขปัญหาดึงคำถามได้แค่ 4 ข้อ ให้กลายเป็น **19 ข้อครบถ้วน** ตามโครงสร้างจริงของเทอมนี้

3. **[Frontend & UI/UX]**
   - **กู้คืนภาษาไทย**: แก้ไขการ Map ข้อมูลจาก `id` -> `questionid` และ `text` -> `question` ทำให้ชื่อคำถามภาษาไทยแสดงผล 100%
   - **ตัวเลือก (Choices)**: ปรับให้ดึง Label จาก `evaluatechoice` (เช่น "มากที่สุด", "น้อยที่สุด") มาแสดงบนปุ่มแทนตัวเลขเฉยๆ
   - **ระบบด่วน (Quick Actions)**: แก้ไขปุ่ม "Tick 5 ทั้งหมด" ให้ทำงานร่วมกับ `questionid` แบบใหม่ได้ถูกต้อง
   - **Validation**: แก้ไขระบบตรวจสอบความครบถ้วนก่อนส่ง ให้รองรับการนับครบ 19 ข้อจริง

4. **[Submission Investigation]**
   - ทดสอบ Payload รูปแบบต่างๆ ของ `Addanswer` และ `commit` ผ่านสคริปต์อัตโนมัติ
   - พบว่าเซิร์ฟเวอร์ตอบรับ `200 OK` แต่ข้อมูลยังไม่ถูกบันทึกถาวรลงฐานข้อมูลหลัก (ยังแสดงผลว่า "ยังไม่ได้ประเมิน")

### 🎯 แผนการดำเนินการปัจจุบัน (Implementation Plan - Current Version)
**เป้าหมายหลัก**: ทำให้การกดส่งผลประเมินจากแอปเรา "มีผลจริง" ในระบบของมหาวิทยาลัย

#### 1. ค้นหา Payload ลับ (Deep Interception)
- **สถานะ**: กำลังดำเนินการ (ติดปัญหา Module Puppeteer ในบางช่วง)
- **แนวทาง**: ใช้ Browser Subagent หรือ Playwright สคริปต์เข้าล็อกอินในเว็บจริง และ "บันทึก" ข้อมูลตัวอย่างเพื่อดู JSON ดิวก่อนถูกเข้ารหัส
- **สิ่งที่ต้องหา**: Key พิเศษใน Payload เช่น `evaluateheadtid`, `remark`, หรือโครงสร้าง Array ของคำตอบที่เซิร์ฟเวอร์ยอมรับการบันทึกจริง

#### 2. อัปเดต API Submit (`api/student/evaluation/submit/route.js`)
- ปรับปรุงให้การส่งข้อมูลมีการจำลองลำดับ (Sequence) เหมือนเว็บจริง
- ตรวจสอบว่าต้องส่ง `Addanswer` ทีละข้อ หรือส่งรวมใน `commit` ทีเดียว (จากผลทดสอบเบื้องต้นคาดว่าต้องส่งรวมใน `commit` ด้วยฟอร์แมตที่เฉพาะเจาะจงกว่าเดิม)

#### 3. การตรวจสอบความถูกต้อง (Final Verification)
- หลังจากส่งจากแอปเราแล้ว จะใช้สคริปต์ตรวจสอบสถานะ `evaluatestatus` ของอาจารย์ท่านนั้นในระบบหลักทันที
- หากเปลี่ยนจาก `0` (รอประเมิน) เป็น `1` (ประเมินแล้ว) = **โปรเจกต์เสร็จสมบูรณ์**
